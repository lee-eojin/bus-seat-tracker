# 수집 운영 기록

## 데이터 경계

- 공개 `bus-seat-tracker`: 수집기 코드와 워크플로만 보관한다.
- 비공개 `bus-seat-tracker-data`: `data/routes`, `data/snapshots`만 보관한다. 당일 수집은 `collect/YYYY-MM-DD` 브랜치에 누적하고, 다음 날 첫 수집에서 날짜당 한 커밋으로 `main`에 반영한다.
- 공개 화면에는 차량 번호, HMAC 차량 ID, 원본 스냅샷을 전달하지 않는다.

## 시작 전 설정

GitHub Actions Secrets에 다음 값을 등록한다.

| 이름 | 역할 |
|---|---|
| `GYEONGGI_BUS_API_KEY` | 공공데이터포털 일반 인증키 |
| `VEHICLE_HASH_SECRET` | 차량 식별자 가명화를 위한 고정 비밀값 |
| `BUS_DATA_REPO_TOKEN` | `bus-seat-tracker-data`에만 Contents 읽기·쓰기가 가능한 fine-grained PAT |

`VEHICLE_HASH_SECRET`은 API 키와 별개로 유지한다. API 키를 재발급해도 같은 차량의 가명 ID가 유지되고, API 키 유출이 과거 가명 ID 대조로 이어지는 일을 막는다.

## 가동 절차

1. `main`에 `collect-bus-seats.yml`을 반영한다.
2. Actions에서 **Collect bus seats**를 수동 실행한다.
3. 비공개 저장소의 당일 `collect/YYYY-MM-DD` 브랜치 `data/snapshots`에 3330·1650 JSONL이 생겼는지 확인한다.
4. 화면에는 관측 시각을 항상 표시한다.

## 수집 스케줄 (2026-07-23 통근 집중 개편)

### 스로틀링 실측이 구조의 근거다

`cron: '*/5'`로 24시간 돌린 실측(2026-07-19~20): 실제 스케줄 실행 간격 **56~105분**, 최대
**4시간 공백**. GitHub은 고빈도 스케줄을 크게 지연시키므로, 크론 간격을 좁혀도 해상도가
확보되지 않는다. 결론 — **크론은 시동 장치로만 쓰고, 해상도는 잡 내부 루프가 담당한다.**

### 창 배치는 만석 분포 실측이 근거다 (2026-07-23)

평일 3일치 관측에서 만석은 통근 창에 집중된다: 06:30 버킷에서 시작(6.0%)해 07:00~08:00에
절정(14.6~17.9%), 저녁은 17:30 시작(8.3%), 18:00~18:30 절정(16.0~18.1%). 21:00 이후는
0.4% → 0으로 사실상 소멸한다. 따라서 통근 창만 집중 수집하고 심야는 수집을 멈춘다.
주말은 피크 없이 균등 스냅샷으로 전환한다 (기존 주말 저녁 루프 폐지).

### 시간대 티어 (KST)

| 시간대 | 창 | 간격 | 방식 |
|---|---|---|---|
| 평일 아침 피크 | 06:30~10:00 | 10분 | 잡 하나가 창 끝까지 루프 |
| 평일 저녁 피크 | 17:30~20:30 | 10분 | 〃 |
| 평일 그 외 | 05:00~21:00 정시 | 1시간 | 매시 크론 1회 스냅샷 |
| 주말 | 06:00~23:00 정시 | 1시간 | 매시 크론 1회 스냅샷 (균등) |
| 심야 | 위 시간대 밖 | — | 수집 없음. 매시 실행은 수집 단계에서 즉시 종료 |

- 시동 크론(UTC): `30 21 * * 0-4`(평일 아침), `30 8 * * 1-5`(평일 저녁), `0 * * * *`(매시 기본)
- **매시 크론이 백업 시동을 겸한다**: 창 시작 60분 전 이후에 도착한 어떤 실행이든 남은 창을
  루프로 이어받는다. 시동 크론이 통째로 스킵돼도 다음 매시 실행이 창을 복구한다.
- 마지막 정시(평일 21시, 주말 23시) 실행이 스로틀로 밀려도 스냅샷을 남기도록 운행 시간대
  판정에 한 시간 여유를 둔다.
- 심야 실행도 잡 자체는 돌므로 전날 브랜치의 `main` 아카이브(자정 이후 첫 실행)는 그대로 동작한다.
- 러너 6시간 한도 보호로 루프는 최대 320분에서 끊고, 잔여 창은 대기 중인 다음 실행이 이어받는다.
- 업로드는 사이클마다 **변경된 파일만** 커밋한다 (체크아웃 시점 파일은 기준선으로 제외).

### 비용·쿼터 (설계값)

- API: 평일 피크 2창 ≈ 39사이클 + 비피크 정시 ~11회 → 기능당 ~100콜/일 (한도 1,000의 10%),
  주말 ~18회 → ~36콜/일
- 러너: 평일 루프 합계 ~6.5시간/일. 공개 저장소라 무료이며, 수집은 이 프로젝트의 목적 자체다.

### 이벤트 모드

콘서트·경기 날에는 Actions에서 **Run workflow**로 `event-duration-minutes`(예: 240),
`event-interval-seconds`(기본 300)를 입력해 수동 시동한다. 시간대 규칙을 무시하고 즉시
지정 시간 동안 지정 간격으로 수집한다.

### 화면 신선도 기준

`prototype-bus/app.ts`의 `expectedIntervalMinutes()`가 위 창 정의와 같은 값을 갖는다.
정상 ≤ 기대주기×2, 경고 ≤ ×4, 초과 시 중단 배너. 심야 휴지와 운행 재개 직후 2시간은
마지막 정시 스냅샷 이후 경과를 기대 주기로 삼아, 밤새 그리고 새벽 첫 스냅샷이 착지하기
전에 배너가 오작동하지 않게 한다. 창 정의를 바꾸면 두 곳을 같이 바꾼다.

## 보관 원칙

원본 JSONL은 비공개 저장소에만 둔다. 공개 배포를 시작할 때는 차량 식별자를 제거한 정류장·시간대 집계만 별도 API로 제공한다.
